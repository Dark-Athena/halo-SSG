背景

基础分析

首先我确认了该环境中，LD_LIBRARY_PATH里已经把GaussDB的libpq配置到了最前面，而且也删除了操作系统里自带的libpq，并且确认了使用的libpq是arm64版本的，和这个服务器是匹配的，但依旧coredump。

由于我本地没有arm环境可以用于复现测试，于是我用了华为云开发者空间的云开发空间里启动了个免费的开发容器，并且在本地的vscode里通过Huawei Developer Space插件连上了这个容器，方便使用AI来排查问题。

细化分析

  • 崩溃链路是：连接失败后在 generators.py 调用 finish_pgconn()，随后在 errors.py 读取 pgconn 属性快照。
  • 这些属性（如 host、password）会走到 pq_ctypes.py，内部通过 _call_bytes() 调 C API（PQhost/PQpass），见 pq_ctypes.py。
  • _call_bytes() 对 c_char_p 返回值会触发 ctypes 解引用；如果 C 层返回的是“非空但无效地址”，Python 来不及抛异常就会直接 SIGSEGV。你看到的 core 正是这一类崩溃。
  • 我做了独立最小复现：在 connect_poll 失败后直接 print(conn.host)，无需上层框架也会立刻段错误，所以问题不在异常对象构造本身，而在“失败态 PGconn + PQhost/PQpass 访问”这个组合。
  • 你 C 程序不崩是因为只调用了 PQstatus/PQerrorMessage/PQfinish，没有触发 PQhost/PQpass 路径。

为什么说是 libpq 组合问题

  • 该驱动在 misc.py 用 find_library("pq") 动态找库，实际加载结果受 LD_LIBRARY_PATH、系统库优先级影响。
  • 你当前这套库报告 PQlibVersion = 90204 (9.2.4)，但又来自 gaussdb-libpq 打包目录，说明是“Gauss 发行库 + 当前系统运行时”的特定组合；在这个组合里，失败态访问某些字符串 getter 不安全。

结论边界

  • 我能 100% 确认“崩溃触发点”是 PQhost/PQpass 在失败态 PGconn 上被调用。
  • 我不能在无 libpq 源码/符号的前提下 100% 断言是 libpq 内部 bug 还是 ABI 兼容性问题，但工程上两者都应按“不安全调用路径”处理（你现在的修复就是正确方向）。

如果你愿意，我可以再给你一版更稳妥的补丁：在 pq_ctypes.PGconn.host/password/... 上加“仅 status == OK 才读”的防护，这样即便别处再调用也不会 core。

我测试了AI中间给的一个临时方案
修改gaussdb的python驱动里的errors.py
把

def finish_pgconn(pgconn: PGconn) -> PGconn:
    args = {}
    for f in fields(FinishedPGconn):
        try:
            args[f.name] = getattr(pgconn, f.name)
        except Exception:
            pass
    pgconn.finish()
    return FinishedPGconn(**args)

改成

def finish_pgconn(pgconn: PGconn) -> PGconn:
    args = {}
    pgconn.finish()
    return FinishedPGconn(**args)

即不读PGconn里的信息，果然不会coredump了。

基于AI的分析再思考

但是我没有直接把这个方案给客户，因为从AI分析的结果来看，是libpq在连接失败的时候，返回了无效地址被python的ctypes读取才导致的问题，而这个python驱动此处的代码和开源的psycopg3里是完全一样的。

这个时候，客户突然反馈，说之前他密码搞错了，把密码改正确后，airflow能正常启动了。。。

我庆幸自己做复现测试的时候也是用了无效的连接地址，才可以马上复现，如果用了个有效连接，我恐怕很难排查到这个问题了。

但是，华为的这个python驱动，其实在github上是有工作流去自动测试的，测试用例里也有连接失败的用例，但这个工作流是跑通了的，而且也跑了arm平台。对比了下，差别在于它工作流用的ibpq是505.2版本的，而我用的是506.0版本的。

然后我马上配好505.2的libpq环境再来跑前面这几个测试程序，都没core了！

到这里，所有的现象综合起来，问题直指ARM64环境中，GaussDB在506.0版本中修改了libpq，导致python驱动在连接失败时会coredump！至于改了哪里，由于没有源码，只能让华为来解释了。我怀着侥幸心理用opengauss的libpq驱动测了下，期望能coredump，这样就有源码来分析，结果并没有coredump。

顺手给华为提了个issue,后续再看吧
