问题根本原因分析

根据您提供的错误信息和堆栈跟踪,我分析出以下问题:

核心问题:

当执行 SELECT 1 FROM sys_dummy CONNECT BY LEVEL<=(SELECT max(a) FROM v1) 时,系统试图打开文件 "base/15830/24718",但该文件不存在。

问题原因:

  1. 视图没有物理文件
  • v1 是一个视图(CREATE VIEW v1 AS SELECT 1 a FROM sys_dummy)
  • 视图在 pg_class 中的 relfilenode 可能是 0 或无效值,因为视图没有对应的物理存储文件
  • 普通表有 relfilenode 指向实际的数据文件,但视图不需要
  1. CONNECT BY LEVEL 中的子查询处理问题
  • CONNECT BY LEVEL 是 openGauss 支持的 Oracle 兼容语法,用于层次查询
    当 LEVEL<= 后面接一个子查询 (SELECT max(a) FROM v1) 时,系统需要先执行这个子查询
    关键问题: 在执行子查询初始化扫描时,系统错误地将视图 v1 当作普通表来处理
    视图展开时机问题
    从堆栈看,执行流程是:
#28 exec_simple_query
#27 PortalStart
#26 ExecutorStart
#22 InitPlan
#19 ExecInitAgg  (处理子查询的聚合)
#16 ExecInitSeqScan
#15 InitScanRelation
#13 BeginScanRelation
#6 InitScanBlocks
#5 RelationGetNumberOfBlocks  -> 调用 smgrnblocks
#3 mdnblocks -> mdopen
#1 mdopenagain -> 报错找不到文件
  • 问题点: 视图应该在 rewrite 阶段通过 fireRIRrules() 展开为其底层查询
  • 但在 CONNECT BY 的子查询中,视图可能没有被正确展开就进入了执行阶段
  • 执行器试图像扫描普通表一样扫描视图,调用 RelationGetNumberOfBlocks()
  • 这导致使用了视图的（无效的）relfilenode 去尝试打开物理文件
  1. START WITH...CONNECT BY 重写机制的问题
  • openGauss 将 START WITH...CONNECT BY 转换为 CTE (Common Table Expression) 递归查询
  • 在这个转换过程中,嵌套在 CONNECT BY LEVEL<= 中的子查询可能没有经过正确的视图重写
    ###为什么会出现 relfilenode = 24718?
    这个 relfilenode 可能是:
  • 视图定义时某种错误的 relfilenode 赋值
  • 或者是某个已删除表的遗留 OID
  • 正常情况下,视图的 relfilenode 应该是 0 或 InvalidOid

建议修复方向:

需要在以下位置添加检查:

  1. 在 InitScanRelation() 或 BeginScanRelation() 中:
// 检查 relkind,如果是视图应该报错
if (relation->rd_rel->relkind == RELKIND_VIEW) {
ereport(ERROR,
errmsg("cannot directly scan view \"%s\"",
RelationGetRelationName(relation)),
errhint("View should have been expanded during query rewrite."));
}
  1. 在 START WITH...CONNECT BY 重写阶段:
    确保子查询中的视图被正确展开,可能需要在 transformStartWith() 函数中检查并处理嵌套的子查询
  2. 在查询重写阶段 (fireRIRrules):
    确保 CONNECT BY 子句中的子查询也经过完整的视图展开处理

这个问题本质上是视图重写和 CONNECT BY 重写之间的交互问题,在特定的SQL结构下(CONNECT BY LEVEL + 子查询 + 视图)触发了边界case。

CONNECT BY LEVEL 子查询视图展开问题修复

问题描述

SQL 场景

CREATE VIEW v1 AS SELECT 1 a FROM sys_dummy;
SELECT 1 FROM sys_dummy CONNECT BY LEVEL<=(SELECT max(a) FROM v1);

###报错信息

ERROR:  could not open file "base/15830/24718": No such file or directory

问题根因

  1. 视图本质:视图(VIEW)在 pg_class 中没有对应的物理文件,其 relfilenode 可能是 0 或无效值

  2. 正常流程: 视图应该在查询重写阶段通过 fireRIRrules() 函数展开为其底层的 SQL 查询

  3. 问题流程:

  • CONNECT BY LEVEL<= 后的子查询 (SELECT max(a) FROM v1) 在 parse 阶段被提取到 connect_by_level_quals

  • 这个子查询在 transformWhereClause() 转换时,没有经过视图展开就进入了执行阶段

  • 执行器初始化扫描时,把视图 v1 当作普通表处理

  • 调用 RelationGetNumberOfBlocks() 尝试打开视图的物理文件

  • 使用无效的 relfilenode 值导致 "could not open file" 错误

  1. 核心问题: START WITH...CONNECT BY 重写机制与视图重写机制的交互存在缺陷

  • CONNECT BY 将查询转换为递归 CTE(Common Table Expression)

  • 在 CTE 的 swoptions->connect_by_level_quals 中的子查询未经过视图展开处理

修复方案

采用两层防护策略:

方案1: 根本性修复 - 在 rewrite 阶段确保视图展开

修改文件: src/gausskernel/optimizer/rewrite/rewriteHandler.cpp

修改位置: fireRIRrules() 函数,处理 CTE 列表的部分

修改内容:

/* Recurse into subqueries in WITH */
foreach (lc, parsetree->cteList) {
   CommonTableExpr* cte = (CommonTableExpr*)lfirst(lc);

   /* 
    * BUGFIX: For START WITH...CONNECT BY converted CTEs, also expand
    * views in connect_by_level_quals and start_with_quals which may
    * contain subqueries referencing views.
    */
   if (cte->swoptions != NULL) {
       if (cte->swoptions->connect_by_level_quals != NULL) {
           (void)expression_tree_walker(cte->swoptions->connect_by_level_quals, 
                                 (bool (*)())fireRIRonSubLink, 
                                 (void*)activeRIRs);
       }
       if (cte->swoptions->start_with_quals != NULL) {
           (void)expression_tree_walker(cte->swoptions->start_with_quals, 
                                 (bool (*)())fireRIRonSubLink, 
                                 (void*)activeRIRs);
       }
   }
   cte->ctequery = (Node*)fireRIRrules((Query*)cte->ctequery, activeRIRs, false);
}

原理说明:

  • fireRIRonSubLink() 会递归处理 SubLink(子查询)中的视图展开
  • 对 START WITH 专有的 swoptions 结构中存储的 quals 也应用视图展开规则
  • 使用 expression_tree_walker() 而不是 query_tree_walker(),因为这些 quals 是 Node* 类型的表达式树
  • 确保 CONNECT BY LEVEL<= 中的子查询里的视图被正确展开

方案2: 防御性检查 - 在执行器阶段捕获异常

修改文件: src/gausskernel/runtime/executor/nodeSeqscan.cpp

修改位置: BeginScanRelation() 函数开头

修改内容:

TableScanDesc BeginScanRelation(SeqScanState* node, Relation relation, 
                               TransactionId relfrozenxid64, int eflags)
{
   Snapshot scanSnap;
   TableScanDesc current_scan_desc = NULL;
   bool isUstoreRel = RelationIsUstoreFormat(relation);

   /*
    * BUGFIX: Catch views that weren't properly expanded during rewrite.
    * This should not happen in normal cases, but provides a clear error message.
    */
   if (relation->rd_rel->relkind == RELKIND_VIEW) {
       ereport(ERROR, (errcode(ERRCODE_FEATURE_NOT_SUPPORTED),
               errmsg("cannot scan view \"%s\" directly", 
                      RelationGetRelationName(relation)),
               errhint("View should have been expanded during query rewrite. "
                      "This may indicate a bug in CONNECT BY processing.")));
   }
   ...
}

原理说明:

  • 即使方案1修复了当前问题,这个检查可以防止未来类似问题

  • 提供更友好的错误提示,而不是混淆的"文件不存在"错误

  • 属于防御性编程的最佳实践

Parse 阶段:

transformSelectStmt()
 -> transformStartWith()
      -> CreateStartWithCTE()
           -> transformWhereClause(context->connectByLevelExpr)  // 转换但未展开视图

Rewrite 阶段:

QueryRewrite()
 -> fireRIRrules()
      -> foreach CTE:
           cte->ctequery = fireRIRrules(cte->ctequery)  // 展开 CTE 主体中的视图
           // BUG: 未处理 cte->swoptions 中的 quals!

修复后的 Rewrite 阶段:

QueryRewrite()
 -> fireRIRrules()
      -> foreach CTE:
           if (cte->swoptions != NULL):
               fireRIRonSubLink(connect_by_level_quals)  // 新增:展开 quals 中的视图
               fireRIRonSubLink(start_with_quals)
           cte->ctequery = fireRIRrules(cte->ctequery)

Execution 阶段 (如果视图未展开会触发):

ExecutorStart()
 -> InitPlan()
      -> ExecInitSeqScan()
           -> InitScanRelation()
                -> BeginScanRelation()
                     // 方案2 的检查在这里触发
                     if (relkind == RELKIND_VIEW) ERROR!

影响范围

受益场景

  • CONNECT BY LEVEL<=(SELECT ... FROM view)
  • START WITH ... IN (SELECT ... FROM view)
  • 任何 START WITH...CONNECT BY 子句中引用视图的子查询

不影响场景

  • 普通的 CONNECT BY 查询(无视图引用)

  • FROM 子句中的视图(原本就会展开)

  • CTE 主体查询中的视图(原本就会展开)
